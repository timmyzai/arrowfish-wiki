开发者上手手册 — 使用 Claude 完成 Sprint 任务的标准流程

更新日期：2026年4月24日 适合：全体开发者（新加入与资深）
📖每个开发者都按同样的 6 阶段流程完成每个任务。读一遍、书签保存、每次都按这个流程做。小团队要能"克隆"标准动作，就靠这份文档。

一、首次配置（仅一次）

  • 把 5 个团队仓库克隆到 ~/Documents/GitLab/skytree/：orebackend · proxy_android2 · flutter_vpn · vpn-internal-reports · .vscode
  • 设置 Git 身份：git config --global user.name "<你的 GitLab 用户名>" + git config --global user.email "<你的邮箱>"
  • 安装 Claude Code CLI，从工作区根目录打开：cd ~/Documents/GitLab/skytree && claude
  • 首次会话运行 /start——查看项目结构和当前 Sprint 状态
  • 读一次：vpn-internal-reports/team-workflow-briefing.md（完整简报）

二、每日开始工作

  • 每个仓库 git pull
  • 从工作区根目录打开 Claude
  • 重命名会话：/rename "<任务简称>"
  • 打开 vpn-internal-reports/Sprint-<N>/README.md 查看 Sprint 状态

三、6 阶段任务流程

┌──────────────────────────────────────────────────┐ │ 每个任务 = 一个 6 阶段循环 │ ├──────────────────────────────────────────────────┤ │ │ │ 1. CLAIM — 从 Sprint 计划中领取未认领任务 │ │ │ │ 2. PLAN — /plan （学习 + 设计 + Sprint 影响检查）│ │ ↳ 如超出 Sprint 容量 → 升级给 Lead│ │ │ │ 3. EXECUTE — /plan execute │ │ │ │ 4. VERIFY — /go-specialist 或 /kotlin-specialist│ │ （只在最终 diff 上运行一次） │ │ │ │ 5. COMMIT — /commit │ │ │ │ 6. CLOSE — /knowledge-base-updater（如有新发现）│ │ /report update │ │ /clear │ │ │ └──────────────────────────────────────────────────┘

四、Phase 1 — 领取任务（CLAIM）

  • 在 Sprint README 里找一个未认领、符合你优先级和技术栈的任务
  • 把该任务的 Owner 改为你的名字，Status 改为 in-progress
  • 从 main 切出分支，前缀按 rules/general/git.md：fea/ 新功能 · enh/ 优化 · fix/ 修复 · ref/ 重构 · hot/ 紧急修复
  • 示例：fix/vpn-kill-switch-doze

五、Phase 2 — 计划（PLAN）

  • 运行 /plan——进入计划模式
  • 让 Claude 探索代码；它问什么问题就如实回答
  • 把模板 templates/task-plan.template.md 复制到 plans/<分支名>.md
  • 填写：背景 · 验收标准 · 方法 · 关键文件 · 风险 · 验证方式 · 不在本次范围 · Sprint 影响
⚠️如果 Sprint 影响评估为"超出 Sprint 容量"：在计划文件加 ## SPRINT IMPACT 章节，写清楚需要多几天 + 可替换任务候选。停下来，不要 ExitPlanMode。在群里告诉 Lead：@lead sprint impact on Task #<N> — +<X>d. Cut-queue: Task #<Y>. Plan at vpn-internal-reports/plans/<file>.md。等 Lead 决策后再继续——绝不自行扩大范围。

六、Phase 3 — 执行（EXECUTE）

  • 运行 /plan execute——Claude 按计划一步步执行
  • 执行期间不要调用其他技能——让 /plan execute 跑完
  • 出现意外情况 → 停下问 Lead，不要偷偷改方向

七、Phase 4 — 验证（VERIFY）

只有当 diff 最终完成、准备提交时才运行。

技术栈验证技能
orebackend / proxybackend/go-specialist double check
proxy_android2/kotlin-specialist double check
flutter_vpn/flutter-specialist double check
跨多栈/code-reviewer
  • 修复 specialist 指出的问题
  • Specialist 只跑一次——不是聊天工具，是检查点

八、Phase 5 — 提交（COMMIT）

  • 运行 /commit
  • 审阅起草的提交信息，批准或调整
  • 标题格式：Add | Fix | Update | Remove | Improve <简短描述>（≤72 字符）
  • 绝对不要用 --amend、--no-verify、Co-Authored-By
  • Hook 失败？修根因 → 新建提交（不要 amend）

九、Phase 6 — 收尾（CLOSE）

  1. 知识库（仅当有非显然的新发现）：/knowledge-base-updater "save: <一句话发现>"。常规工作跳过。
  2. 执行报告（每次必做）：复制 templates/execution-report-entry.template.md，追加到 Sprint-<N>/execution-reports/phase-<name>.md。填写：完成了什么 · 非显然发现 · 后续任务 · Sprint 影响。
  3. 更新 Sprint README：把你任务的 Status 改为 done，提交到 vpn-internal-reports。
  4. 在产品仓库发起 MR，指派 Lead 审核。
  5. /clear——释放上下文，任务结束。

十、角色分工（Lead 与 Dev）

动作LeadDev

设置优先级 / 估计天数✅❌
认领未分配任务✅（如亲自编码）✅

执行 + 提交 + 推送✅✅

十一、日常铁律（永远不要破坏）

  • 一次一个技能——不要把 /plan + /plan execute + /commit 堆在一起
  • 一个会话 = 一个任务——任务之间 /clear
  • 不准自行扩大范围——发现任务变大了，升级给 Lead
  • Specialist 仅 Phase 4——不是"帮我看看方案"的聊天工具
  • 永远用模板——task-plan 和 execution-report 模板，不要自由发挥
  • 一任务一 MR——Lead 审核通过才合并

十二、出问题怎么办

情况处理方式
Build 失败当前会话修复，重新跑 specialist，修好才能提交
提交 hook 阻止修根因，新建提交，永远不用 --no-verify
Merge 冲突本地解决，在解决后的 diff 上重新跑 specialist
远程有新提交git stash -u → git pull --ff-only → git stash pop → 重新 build
会话乱了、没有计划/clear，从 Phase 2 重新开始

十五、详细教学 — 每个技能到底怎么用

按真实场景演示，照着做就行。每个技能只展示最常见的用法，不堆砌冷门选项。

15.1 /plan — 开始计划一个任务

什么时候用：新任务开始。目标：让 Claude 帮你把"要做什么"想清楚，写到计划文件里。

你：/plan
Claude：(进入计划模式，可能派出 Explore 子代理扫代码)
Claude：我看到你想修 kill switch 在 Doze 下失效的 bug。
        几个问题先确认：
        1. 只改 Android 还是也要改 iOS？
        2. 你希望 kill switch 在 Doze 下保持锁定，还是允许用户手动释放？
        3. 之前有没有测过 Android 13+ 的新 Doze 规则？
你：只改 Android；保持锁定；没测过 13+
Claude：好的，写计划到 ~/.claude/plans/abc-xyz.md
        (列出方法 / 关键文件 / 风险 / 验证方式)
你：(检查计划文件，填入或补充 acceptance criteria)
你：看起来 OK，可以进入 execute
Claude：ExitPlanMode
💡关键点：Claude 问问题时如实回答，不要敷衍。计划错了，后面全错。

15.2 /plan execute — 按计划执行

什么时候用：计划已批准，准备写代码。

你：/plan execute
Claude：开始 Phase 1 的第一个逻辑组...
        (Claude 按计划一步步做：学习 → 实现 → 验证 → 提交 → 报告 → 更新索引)
Claude：Phase 1 完成，要不要继续 Phase 2？
你：继续
...
⚠️不要在 /plan execute 跑的时候调其他技能（/go-specialist、/commit 等）。让它跑完一整组再说。

15.3 /go-specialist double check（或 kotlin / flutter）— 验证

什么时候用：代码改好了、build 过了、准备提交之前。只运行一次。

你：/go-specialist double check
Claude：(读 git diff，跑 go build，检查规范)
Claude：发现 3 个问题：
        1. handler/client/user.go:305 — 应该用 MDBSlave() 而不是 MDB()
        2. mod/models/user_device.go:40 — 错误没有用 %w 包装
        3. handler/client/init.go — gofmt 告警
你：(修这 3 个问题)
你：/go-specialist double check
Claude：所有问题已解决，build 通过，可以提交。
🚫反模式：把 specialist 当成"帮我想方案"的聊天工具。它是检查点，不是设计伙伴。设计问题用普通 prompt 或 /plan 阶段解决。

15.4 /commit — 提交 + 推送

你：/commit
Claude：起草提交信息：
        "Fix VPN kill switch Android Doze failure"
        (显示所有 staged 文件列表)
你：批准
Claude：git commit && git push origin HEAD
Claude：提交 1e23b3a 已推送到 origin。
⚠️如果 pre-commit hook 失败：修根因（不是 --no-verify），然后新建提交（不是 --amend）。

15.5 /knowledge-base-updater — 保存非显然发现

只有在你学到了代码里看不出来的东西才用。例如：

  • 某个线上客户端依赖某响应字段，删了会无限循环
  • 某错误码有副作用（比如触发重新登录弹窗）
  • 某 Redis key 的 TTL 比文档写的短

不要用来存"我改了这个函数"之类的常规信息——代码和 commit message 就是文档。

你：/knowledge-base-updater "save: skytree_pc v2.2.71 always sends nonce to /user/auto-register; backend must keep nonce validation optional-when-present"
Claude：(确认要写到哪个知识库文件，展示 diff)
你：批准
Claude：已写入 .claude/knowledge/auth-passwordless-design.md

15.6 /report update — 写执行报告

每任务必做。即使是 trivial 的修改。让后来人知道你做了什么。

你：/report update
Claude：这次 Sprint 是 Sprint-2。最近的 phase 是 phase-android-stability-refactor。
        要追加新条目吗？
你：新建 phase-auth-register-rewrite
Claude：(按 template 生成条目)
        ## Task #3 — Collapse register flow to single endpoint (2026-04-24)
        Owner: Tein Fai
        Commits: orebackend 9786b9eac, proxy_android2 cd3c925d
        What shipped: ...
        Non-obvious findings: ...
        Follow-ups: ...
        Sprint impact: 0 days (fit)
你：批准
Claude：已追加。

15.7 /clear — 清上下文

任务结束后必做。把当前会话的上下文全部释放，下个任务从干净的上下文开始。保证 token 不浪费、不串任务。

你：/clear
Claude：上下文已清空。准备好接下一个任务了。

15.8 逃生出口 — 会话乱了怎么办

以下三种情况，直接 /clear 重来，不要继续撑：

  1. 同一个会话开了好几个无关话题
  2. 修了好几处但已经不记得哪些改对了
  3. Claude 的回答开始重复或矛盾

重来一次 10 分钟；硬撑 1 小时代价更大。

十六、团队改进建议通道

🗣️这份流程不是刻在石头上的。团队每个人都欢迎提改进建议——在实际使用中发现哪里别扭、哪里浪费时间、哪里规则说不通，直接提出来一起优化。

你可以随时反馈

发现什么怎么做
某个技能用起来卡、报错、表现不符合预期在群里 @Lead + 代码仓库贴出来。如果很急，先用 /contribute-rules 记录。
某个阶段的规则不合理（比如"specialist 只跑一次"在某场景下做不到）在周会提出，或先写一个 counter-example 发群里
某个模板缺字段 / 多余字段直接改模板，发 MR 到 vpn-internal-reports/templates/，Lead 审核合并
某个知识点应该加进知识库/knowledge-base-updater 直接写，或群里告诉 Lead 由他补
某个技能应该增加新功能 / 删掉冗余用 /contribute-rules 起草改动，MR 到 .claude/skills/ 对应技能

Sprint 回顾（每个 Sprint 结束）

每次 Sprint 结束，Lead 会召开 15 分钟回顾会，三个问题：

  1. 什么流程卡住你了？——哪些阶段浪费时间、哪些规则反人类
  2. 什么发现应该写进团队文档？——这个 sprint 学到的非显然知识
  3. 下个 Sprint 有什么建议？——技能 / 模板 / 规则 / 知识库的改进

反馈直接导致下一个 Sprint 的 .claude/ 更新。每人的意见都算数。

匿名 / 不好意思公开说的

任何时候，直接私聊 Lead。特别是以下情况请直接说：

  • 你对流程感到困惑但不好意思在群里问（100% 是流程写得不清楚，不是你问题）
  • 你觉得某条规则会害你出错，但不确定怎么改
  • 你发现其他同事在绕过流程（先不要点名，描述现象即可）

文档改进原则

  • 改文档就是改流程——文档是真的 source of truth，不是摆设
  • 小改动直接 MR——别开会讨论 typo
  • 大改动（删规则、加阶段）——先群里讨论，共识后再改
  • 改完要告诉团队——Slack 一条消息："我把 XX 改成 YY，原因是 ZZ"
🤝这份流程属于团队。不是 Lead 一个人定的、不是 Claude 一个工具的规矩。每次有人用起来觉得别扭，就是流程还能更好的信号。说出来，一起改。

AI 使用透明度 — 每周导出用量

📊每个人每周导出一次自己的 AI 使用数据，分享到团队频道。不是考勤打卡——是让大家都能看到"一个任务应该花多少 token、哪种操作最贵"，集体学会更高效地用 Claude。

怎么导出

  • 在当前会话（或任意会话）运行：/cost——显示该会话的 token 用量 + 成本
  • 如需跨会话汇总，参考 Claude Code 的 ~/.claude/projects/ 下的会话日志
  • （如果团队后续做了自定义的 /insights 聚合脚本，直接用它）

每周分享什么（发到 Slack / 群）

字段示例为什么分享
本周任务数3 个看 throughput
总 token 消耗2.1M估算成本
平均每任务 token~700K看是否偏高——可能流程没跑好
最贵的一次是什么Task #5：一个会话里连调了 4 个技能反模式早期发现
最省的一次是什么Task #2：按 6 阶段跑、每阶段 clearPattern 推广
一个发现"spawn Explore agent 比 /plan 贵很多"沉淀到知识库

为什么要做这个

  • token 就是钱——Opus 一次会话轻松上 100 万 token。团队年支出不小。
  • 量化 = 改进——看不到数字就不知道哪里浪费。
  • 同辈学习——新人看老人怎么用最省 token，成长最快。
  • 找 anti-pattern——如果全组都花大，说明流程本身需要改（不是个人问题）。

匿名化原则

分享数据是团队成长工具，不是绩效考核。Lead 不会根据 token 用量评价个人。重点是发现流程问题、传授最佳实践。

⚠️反过来也要说："少用 token" 不等于"少用 Claude"。一次深入的 /plan 比 10 次零散问答有价值得多。目标是有意识地花，不是吝啬。
