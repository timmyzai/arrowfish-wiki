节点运维手册

VPN 节点生命周期、自动化能力清单、运维 SOP — 内部团队参考

适合：开发团队、运维 · 更新于：2026 年 5 月

1. 节点生命周期一览

从单个 VPN 节点诞生到退役的完整路径，标注每一步的自动化程度：

阶段动作自动化状态
1. 采购从 Lisahost / iPRaft / GCore 等供应商手动购买 VM手动
2. 注册到 BOBO 创建节点记录或通过 /agent/register-or-heal 自识别（按 main_ip）自动
3. DNS 子域kebab → Provision DNS → 自动选下一个 {region}{seq:02d}.{zone}自动 (Phase B)
4. Bootstrap 部署BO 生成一次性引导 URL，节点 SSH 执行脚本，安装 sing-box + agent半自动
5. 配置同步agent 每 60s 拉取用户列表 + 上游配置自动
6. 健康监控NodeHealthReconcileJob 每分钟跑状态机：active → degraded → billing_dead / temp_down → active自动 (Phase 2)
7. 自动下线billing_dead / temp_down / 流量超阈值 → 自动从线路池移除 + 一条 TG自动 (Phase A)
8. 自动恢复双信号稳定 2 分钟 → 自动还原线路池 + 一条 TG自动
9. 续费 / 标记已付kebab → Mark paid + retry → 立即重探 + 推后 30 天到期自动 (Phase 2)
10. DNS 释放（节点退役）kebab → Release DNS → "保留以备复用" 进入释放池自动 (Phase B)
11. 孤立 DNS 清理BO 干跑预览 → 勾选 → Apply。强制干跑优先自动 (Phase B)
12. 整批升级 agentBO 设置 target_agent_version + 维护时间窗，agent 自更新自动 (Phase 2)

2. 节点健康状态机

每个节点 (node_hosts 表 1 行 = 1 台 VM) 都有一个 status：

状态含义触发条件线路池影响
active正常服务—在线
degraded心跳沉默 5 分钟以上 OR TCP 探测失败开始诊断暂未移除
provider_temp_down探测失败 + 付费日未到判定为临时故障移除
billing_dead探测失败 + 付费日已过 (或宽限 2h 后未知)欠费下线移除
unknown_dead已知死亡但原因不明等待管理员判断移除
reconciling整批重装进行中BO 触发移除

所有阈值都在 设置 → 节点健康 / DNS 自动化 调整，无需重启后端。

3. 后台管理介面

3.1 节点列表 — kebab 操作菜单

每行一个 ⋮ 按钮，包含 11 个操作（已分组）：

  • 查看：详情 / 在线用户 / 部署日志
  • 管理：编辑 / 引导 URL / IP 池 / Mark Paid / 重装 Agent / 申请 DNS / 释放 DNS
  • 状态：暂停 / 恢复
  • 危险：软删除（红字）

3.2 设置 → 节点健康 / DNS 自动化

  • 健康阈值：心跳超时、探测超时、未知付费日宽限、恢复稳定窗口、临时故障告警门槛、巡检间隔（共 8 项整数）
  • 自动下线：流量阈值百分比（默认 95%）
  • DNS：GoDaddy Key/Secret、Cloudflare Token（默认隐藏，眼睛图标可显示）、托管 zone CSV、子域命名模板
  • API Key 留空 = DNS 自动化关闭。每个 Key 旁有 测试连通 按钮

3.3 节点管理 → DNS 孤立记录清理

  1. 页面打开自动跑干跑
  2. 列出每个 zone 的孤立记录（DB 中无、或标记为 orphan）
  3. 勾选要删除的，点 应用所选，确认后才真正删除
  4. 已释放（released）的子域永远不会被清理触碰

4. 子域命名规范

默认模板：{region}{seq:02d}.{zone}，例如：

  • jp01.oportal.cc — 第 1 个日本节点
  • hk03.oportal.cc — 第 3 个香港节点
  • us2002.oportal.cc — 4 位序号也支持（模板可改为 {seq:04d}）

子域复用规则（按地区严格）

退役节点若 Release 时选 保留以备复用，子域进入释放池：

  • 新节点 → Provision DNS 时显示 「自动生成新名称」 和 「从已释放池复用」 两个 tab
  • 只能复用同地区子域（jp 释放池只能给 jp 新节点用）
  • 跨地区复用 = 手动 SQL 修改（不允许）

5. Telegram 告警规则（每个事件最多一条）

合并到现有 专线付款 频道。仅以下事件会发：

  1. 节点进入 BILLING_DEAD：「{host} 移除：billing_dead — 影响 N 条线路。请支付发票并在 BO 点 Mark Paid」
  2. 节点进入 PROVIDER_TEMP_DOWN（持续 30 分钟未恢复）：提示可能需要联系供应商
  3. 节点进入 UNKNOWN_DEAD（2h 未知）：请管理员确认
  4. 恢复 → ACTIVE：「{host} 已恢复 — N 条线路重新上线」
  5. 抖动（1h 内 ≥ 3 次状态切换）：一条总结性告警

不会发送：5 分钟级别的心跳偶发、单次探测失败、节点常态 active 状态。

6. 部署 SOP（生产首次启用）

  1. fea/node-driven-quota 经 MR review 后并入 main
  2. CI/CD 自动应用所有迁移到 Staging
  3. Day 1：验 node_hosts 行随 agent 注册成功；vpn_node_usages 有数据；NodeHealthReconcileJob 不会乱标
  4. Day 2：选 1 台 staging 节点 BO 整批重装 → 观察 24h
  5. Day 3-5：持续观察 TG 告警；如阈值不合适在 BO 调整（不需重启后端）
  6. Day 5+：打 production tag，CI/CD 部署到生产；维护时间窗内整批重装生产节点
回滚路径：每个迁移都是增量（只 ADD COLUMN，没有 DROP）。回滚代码即可恢复原行为，新表保留但无人使用。所有新设置默认为 OFF 或安全值。
